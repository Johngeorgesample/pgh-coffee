1. **`/api/shops/report-amenities`** — no test at all, and unlike `/api/shops/report`
   it does **not** validate that `shop_id` exists, and accepts an arbitrary
   `amenities` array with no validation against the known amenity vocabulary. This
   is the corrections/amenities submission flow the brief calls out specifically.
2. **`IssueForm`** (corrections form UI) — no test. Also: on a failed/erroring
   submit it only `console.error`s — the user sees no feedback at all (button just
   stops spinning).
3. **`AmenityReportModal`** (amenities tagging UI) — no test, same silent-failure
   issue as IssueForm.
4. **Account `Favorites`** component (lists/favorites feature) — no test. Also: a
   fetch *error* is treated identically to "you have zero favorites" — misleading
   if the API call fails.
5. **`/api/events` and `/api/events/[eventId]`** — no tests for the events feature.
6. **`/api/curated-lists`** — no test.
7. **`/api/auth/event` and `/app/auth/callback`** — no tests for these auth-boundary
   routes (session exchange, sign-in/up/out telemetry).
8. **No Next.js `error.tsx` boundaries anywhere** — any unexpected render error in
   `app/` or `app/account/` falls through to Next's default error page with no
   app-specific recovery UI.
